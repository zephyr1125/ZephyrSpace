# Watchlist 分层规则

> **重要：任何 AI 在修改 watchlist 文件之前，必须先完整阅读本文件。**

本文件是 watchlist 分拆文件（`watchlist_meta.json` / `watchlist_core.json` / `watchlist_growth.json` / `watchlist_radar.json`）的配套规则说明，定义了三档 watchlist 的分层标准、进入/退出条件和操作约定。

---

## 整体原则

- **只收录已完成完整 PreBuy 分析的公司**（对应 `01-公司/` 下存在完整模板页面）。
- 三档是确信度和操作意图的分层，不是市值或行业的分类。
- 每次新增公司，**必须先判断档位再填写字段**，不允许"先放 radar 再看"的懒惰做法。
- 价格区间字段（`price_bands`）是触发操作的核心，必须根据 PreBuy 页的结论填写，不得留空。
- **档位（core/growth/radar）由公司基本面质量决定，与当前股价无关。** 价格仅影响 `price_bands` 的操作区间——即"何时买"，而非"是否进入 watchlist 或进入哪个档位"。好公司在价格高时同样应留在 watchlist，以便价格回落时能第一时间识别机会。

---

## 三档定义

### 🟢 核心池 `core`

**定位**：买了就放，长期底仓。护城河足够深，价格合理时可主动建仓，无需等待特定催化剂。

**进入条件（需同时满足）**：

| 维度 | 标准 |
|---|---|
| 护城河 | 行业垄断、品牌定价权、或不可复制的资产（水电/核电/牌照），竞争者难以动摇其地位 |
| 盈利质量 | 净现比 > 90%（最近完整财年），ROE > 15% 且过去 3 年持续稳定 |
| 成长要求 | 不需要高增长，稳定或缓慢增长即可；但不能出现结构性利润下滑 |
| 分红 | 优选有稳定分红记录；股息率 > 2% 或承诺分红率 > 40% 者加分 |
| 红旗数量 | PreBuy 报告中主要红旗 ≤ 2 条，且无结构性/治理类红旗 |

> ⚠️ **当前股价不是 core 的进入条件。** 即使公司现在估值偏高，只要基本面满足上表，仍应放入 core。价格高低反映在 `price_bands`：追高区不买，等价格落入较好区间或更低区间再行动。

**操作含义**：价格落入"较好区间"时可主动建仓，不等更多验证；价格落入"更低区间"时加仓；追高区坚决不买。

**退出条件**（以下任一触发则降档至 growth 或移出）：
- 护城河出现结构性损伤（竞争对手大规模侵蚀核心市场）
- 连续 2 年净现比 < 80%（现金流质量恶化）
- ROE 持续下行至 10% 以下
- 出现治理红旗（大股东减持、财务造假迹象、管理层集中离职）

---

### 🔵 成长池 `growth`

**定位**：高质量成长公司，有清晰的成长驱动力，但护城河尚未达到垄断级别，或有关键成长节点仍待业绩验证。等价格窗口，不追高。

**进入条件（需同时满足）**：

| 维度 | 标准 |
|---|---|
| 基本面质量 | 净现比 > 80%，ROE > 15%（或处于上升趋势中） |
| 成长逻辑 | 有清晰的中长期成长驱动力（行业空间 + 公司执行力），已有历史业绩印证方向正确 |
| 护城河 | 有明确的差异化壁垒（技术、品牌、渠道、规模），不需要垄断级别，但需要可识别 |
| 红旗数量 | PreBuy 报告中主要红旗 ≤ 3 条，且红旗均为"可跟踪可验证"类，无根本性治理问题 |

> ⚠️ **当前股价不是 growth 的进入条件，也不是与 core 的区分标准。** core 和 growth 的区别在于护城河深度和成长逻辑的验证程度，而非当前估值高低。无论公司现在贵还是便宜，根据基本面质量判断档位；价格区间信息记录在 `price_bands` 中。

**操作含义**：等价格进入"较好区间"甚至"更低区间"时再买，不追高；如果有明确的业绩验证节点（如季报超预期），可在中性区小仓参与。

**退出条件**：
- 升档至 core：估值回落至历史合理区间，且护城河进一步夯实，可调整至 core
- 降档至 radar：出现基本面恶化信号（净利下滑超 20% 且非一次性、毛利率持续下行、现金流恶化），或成长逻辑被证伪
- 移出 watchlist：基本面持续恶化超过 2 个季度，原有逻辑不再成立

---

### 🟡 雷达池 `radar`

**定位**：有关注价值，但存在较多不确定性。只跟踪，不主动买入。出现明确信号后再考虑是否升档。

**进入条件（三层，缺一不可）**：

**前置质量门槛（新增，优先判断）**

> ⚠️ **必须先通过粗筛才有资格进入 Radar。** 粗筛未通过（ROE 低于行业门槛×80%）的公司不入 watchlist 任何档位，只记录在对应指数专题页的成分股表中。
>
> 粗筛边界（⚠️）内的公司（ROE 在行业门槛 80%~100%，即偏离≤20%），**额外需满足两项**：
> 1. `entry_trigger` 中必须包含明确的 ROE 相关改善条件（净利率回升、ROE 绝对值达到某水平等），**不允许仅凭价格触发**；
> 2. ROE 必须处于**稳定或改善趋势**。若 ROE 连续 2 年持续下行且无明确拐点，则视为结构性退化，**不入 watchlist**，仅记录指数专题页。

**第一层：满足以下任一情形**

| 情形 | 说明 |
|---|---|
| 逻辑未完全验证 | 成长故事有说服力，但缺乏足够的历史业绩印证；或处于行业周期底部，反转拐点不明 |
| 红旗较多 | PreBuy 报告中有 ≥ 3 条主要红旗，或有 1 条治理/结构性红旗 |
| 竞争格局不清 | 行业竞争激烈，公司护城河存疑，胜出概率尚不确定 |
| 主题纯度工具 | 用于跟踪特定行业主题景气度，但公司自身不是该主题的最优受益者 |
| 困境反转型 | 当前业绩较差，押注未来改善，不确定性高 |

**第二层：必须同时具备至少一个具体的、可量化的入场触发器**

> 触发器示例：股价跌至 X 元（对应 DY≥Y% 或 PE≤Zx）、某季报净利同比由负转正、某产能/项目投产落地、行业政策明确落地。
>
> **不符合要求的触发器**："等基本面改善"、"等市场情绪好转"——这类描述过于模糊，不具操作性。
>
> **没有具体触发器的公司，记录在对应指数专题页的成分股表中即可，不进入 watchlist。**

触发器写入 JSON 的 `entry_trigger` 字段（**radar 专属必填字段**），格式为一句话，包含具体条件和数字。

**操作含义**：只读不买。持续跟踪季报和行业动态；触发器条件达成后评估是否升档；若逻辑持续验证（连续 2 个季度超预期 + 红旗消除），升档至 growth；若逻辑证伪，移出 watchlist。

**退出条件**：
- 升档至 growth：核心红旗消除，成长逻辑获得业绩印证，可升至 growth
- 移出 watchlist：跟踪满 2 年无改善信号，或原有关注理由不再成立

---

### 🔴 不入出口 `excluded`

**定位**：完成了 PreBuy 分析，但存在结构性缺陷，不具备进入 watchlist 任何档位的条件。记录结论供备查，不写入 JSON。

**判定标准（满足以下任意 2 条即直接排除）**：

| 标准 | 说明 | 典型例子 |
|---|---|---|
| ① OCF 与利润严重背离 | 净利润正增长但经营现金流持续背离（OCF/净利 < 20%，或非经常性损益占净利 > 60%） | 中科蓝讯：11.5亿非经常性损益使表观净利暴增，OCF/净利仅4%（背离率96%） |
| ② 无明确有时限的修复路径 | `entry_trigger` 本质是"等主业好转"——无法确定扣非净利/现金流何时恢复；若修复时间线超过 3 年或完全不可预期则触发 | 中科蓝讯：TWS市场饱和+主业增速+1.2%，无法预期扣非何时改善 |
| ③ 结构性治理或合规风险 | 外资母公司持股 > 50% 且叠加出口管制风险；或审计意见非标；或存在持续关联交易掩盖利润转移 | 盛美上海：ACMR持股74.5% + 美国出口管制双重叠加 |

**与 radar 的区别**：
- radar 的公司：有明确、可量化的入场触发器，等信号达成再评估
- 不入的公司：结构性问题无法通过等待或价格下跌来规避，进入任何档位都不合适

**操作说明**：
1. 在对应公司页面的 `## PreBuy 结论` 中注明"**不入 watchlist**，原因：[触发的标准+具体数据]"
2. 在对应指数专题页的成分股表中，将该公司状态标记为"❌ 不入"
3. **不写入** `stock_watchlist.json`

## 分层决策树

收到一家新公司的 PreBuy 报告后，按以下步骤判断档位：

```
1. 是否完成完整 PreBuy 分析？
   → 否 → 不得加入 watchlist

2. 是否触发"不入"出口？（满足以下任意 2 条）
   ① OCF/净利 < 20% 且非经常性损益 > 60% 净利
   ② 无明确有时限修复路径（修复时间 > 3 年或完全不可预期）
   ③ 结构性治理/合规风险（外资持股>50%+出口管制叠加、非标审计等）
   → 触发 2 条及以上 → ❌ 不入 watchlist（在公司页和指数专题页注明，不写入 JSON）

3. PreBuy 结论是否支持"可以布局/可以买"（而非"纯观察"）？
   → 否（纯观察/逻辑未验证） → 先判断是否有具体入场触发器；有 → radar；无 → 只记录在指数专题页，不进 watchlist

4. 护城河是否为垄断/极强级别？净现比 > 90%？ROE 稳定 > 15%？红旗 ≤ 2？
   → 是 → core

5. 基本面优秀但估值偏贵，或有成长验证点待确认？净现比 > 80%？ROE > 15%？红旗 ≤ 3 且无治理问题？
   → 是 → growth

6. 以上都不满足（红旗多/逻辑存疑/周期不明/纯主题工具）
   → 是否有具体可量化的入场触发器（价格/季报/政策）？
      有 → radar（必须填写 entry_trigger 字段）
      无 → 不进 watchlist，记录在指数专题页成分股表即可
```

## 字段填写规范

新增一条 watchlist 记录时，以下字段不得省略：

| 字段 | 要求 |
|---|---|
| `name` | 公司简称，与 `01-公司/` 页面标题一致 |
| `code` | 必填且不得为空。A 股使用 `XXXXXX.SH` / `XXXXXX.SZ` / `XXXXXX.BJ`；美股使用 `TICKER.US`，例如 `AAPL.US`、`GOOGL.US`；港股使用 `XXXXX.HK` |
| `source_etf` | 来自哪个指数/ETF 的研究，填指数代码；若非指数来源填 `"direct"` |
| `position_role` | 一句话说明在组合中的角色定位（底仓/成长弹性/周期工具等） |
| `prebuy_conclusion` | 直接引用 PreBuy 页面的操作结论，不得自行概括 |
| `watch_reason` | 为什么跟踪这家公司（区别于 prebuy_conclusion，这是长期关注理由） |
| `current_price` | 加入时的当前价格（数字，单位元） |
| `price_date` | `YYYY-MM-DD` 格式，表示 `current_price` 的行情日期 |
| `price_bands` | **固定 3 个数字的降序数组** `[avoid_min, watch_min, consider_min]`。label 和 action 由索引位置隐式确定（见下表），不写入 JSON，以压缩文件大小。 |

**price_bands 索引与语义对照**（固定，不得修改）：

| 索引 | 档位名 | action | 价格区间 | 含义 |
|---|---|---|---|---|
| `[0]` | 追高区 | `avoid` | price ≥ bands[0] | 明显高于合理上沿，赔率差，不买 |
| `[1]` | 中性区 | `watch` | bands[1] ≤ price < bands[0] | 逻辑对但不便宜，观察持有 |
| `[2]` | 较好区间 | `consider` | bands[2] ≤ price < bands[1] | 赔率开始顺手，可研究性布局 |
| 隐含第4档 | 更低区间 | `strong_buy` | price < bands[2] | 安全边际更好，可积极加仓 |

**price_bands 写法示例**（正确）：

```json
"price_bands": [85.0, 68.0, 55.0]
```

解读：price ≥ 85 → 追高区(avoid)；68 ≤ price < 85 → 中性区(watch)；55 ≤ price < 68 → 较好区间(consider)；price < 55 → 更低区间(strong_buy)。三值降序排列，结构上不可能产生 gap，禁止使用对象格式或增减元素数量。

**valuation_anchor 写法规范**：

`valuation_anchor` 是 `price_bands` 的文字说明，**必须与 price_bands 同时填写**，记录"这四档价位是按什么逻辑定出来的"，供后续 AI 或人工核查时直接对着锚更新数字。

格式模板：
```
"valuation_anchor": "估值锚类型：追高区>X(说明) / 中性区Y-X(说明) / 较好区间Z-Y(说明) / 更低区间<Z(说明)"
```

不同公司类型的估值锚选择：
- **成长股**：前瞻 PE / PEG / EV/EBITDA + 增长验证节点
- **周期股**：利润中枢 × 历史 PE 区间 + 景气位置
- **价值/底仓股**：历史 PE 区间 / PB / 股息率 + 安全边际
- **困境反转股**：修复进度（扣非 / 现金流）× P/S 或 PB

| `risk_flags` | 数组，至少 2 条，直接引用 PreBuy 页面的主要红旗 |
| `board` | 单字：`沪`=上交所主板 / `深`=深交所主板 / `创`=深交所创业板 / `科`=科创板 / `北`=北交所 / `纳`=NASDAQ / `纽`=NYSE |
| `market` | 可选字段，仅美股/港股填写：`"US"` 或 `"HK"`；A 股不填 |
| `valuation_anchor` | 必填，记录四档价位的估值逻辑，格式见上 |
| `dv_ttm` | 股息率（DV TTM，单位 %，数字类型）；来自 tushare `daily_basic` 最新一条；`null` 代表暂无股息或数据缺失；定期随行情更新 |

**字段一致性硬规则**：

- watchlist 记录必须使用同一套标准字段：`name`、`code`、`source_etf`、`position_role`、`prebuy_conclusion`、`watch_reason`、`current_price`、`price_date`、`board`、`price_bands`、`risk_flags`、`valuation_anchor`、`dv_ttm`。
- `radar` 档额外必须有 `entry_trigger`。
- 美股/港股可额外有 `market`，美股固定为 `"US"`，港股固定为 `"HK"`。
- 禁止把中间筛选字段直接写入最终 JSON，例如 `ts_code`、`sector`、`tier`、`thesis`、`pe_range`、`roe_ttm`、`revenue_growth`、`market_cap_bn`、`notes`。
- 若需要保留这些信息，必须合并进 `position_role`、`watch_reason`、`prebuy_conclusion`、`risk_flags` 或 `valuation_anchor`。
- `code` 禁止为空字符串或缺失；自动化脚本会以 `code` 作为唯一键，空值会造成 key 冲突。
- 美股 `code` 必须带 `.US` 后缀，供行情脚本路由到 yfinance 通道；不得写裸 ticker，例如 `AAPL`、`GOOGL`、`AMZN`。

## 现有分档说明

截至本规则建立时（2026-04-26），现有公司分布：

- **core（核心池）**：13 家，垄断级护城河+净现比>90%+ROE稳定>15%+红旗≤2
- **growth（成长池）**：42 家，基本面优秀、成长逻辑清晰的公司
- **radar（雷达池）**：21 家，需要更多验证或有明显红旗但具备明确入场触发器的公司

完整列表见 `data/stock_watchlist.json`。
